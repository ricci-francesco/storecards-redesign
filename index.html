StoreCard Anatomy

Card Types & Block Structure

StoreCards come in three sizes. Each defines a fixed set of generic blocks — containers with a position and a type. The backend assigns modules to blocks at render time based on context configuration.

Dimension 1
Surface
Which screen of the app the StoreCard is rendered on.
StoreWall Search DiscoveryWall
Dimension 2
Vertical
The business category of the stores shown in that surface.
Food Groceries Retail
Layout unit
Block
A named, typed container at a fixed position inside the card. The block defines where and what kind of content can appear — not what the content is.
Content unit
Module
A backend-resolved content unit (Highlights, TrustMarkers, Promotions…) assigned to a block by the config. The module brings its own hierarchy logic.

Block reference
A Tag — top-left overlay / info-row
Primary tag slot. On Large: image overlay top-left. On Medium/Small: info-row above store name. Accepts: Highlights, TrustMarkers.
B Tag — image overlay stacked
Large only. Secondary tag stacked below A on the image. Accepts: TrustMarkers, Highlights.
C Label — image bottom-left
Large only. Label overlaid bottom-left of the image. Accepts: store category, distance, or custom label.
D Pagination — image bottom-right
Large only. Dot pagination indicator for image carousels. Shown when store has multiple images.
E Tag — below Core Info
Large only. Dedicated row below Core Info. Accepts: Promotions, TrustMarkers. Exp 1 tests moving Promotions here.
B (Med) Images — below top row
Medium only. Horizontal image strip below the logo+info row. Accepts: TopSellers carousel or StorecardImage. Absent in Small.

Highlights

Trust Markers
Trust Markers
Large → Block B  ·  Medium/Small → Block A (when Highlights OFF)  ·  Elements: FriendsOrders, MostPopular, InStorePricing, FreshnessGuarantee
Promotions
Promotions
Large → Block E (current config) or Block A (Exp 1 control: was overlay)  ·  Medium → not shown  ·  Elements: Promotion tag
Features
Features
Large → Store image fills image area (no block)  ·  Medium → Block B (TopSellers carousel)  ·  Small → absent
Core Info
Core Info
All card types — fixed area below image (Large) or right of logo (Medium/Small). Always ON, no block assignment needed.

Experiment 3

Most Popular in Cuisine

Introduce a new cuisine-scoped TrustMarker — "Most Popular in [Cuisine]" — awarded to the store with the highest order volume within its cuisine category. Test whether showing it lifts CTR vs. the current TrustMarkers hierarchy.

New TrustMarker — eligibility & logic
Signal
MostPopularInCuisine
Awarded to the #1 store by order volume within a cuisine (e.g. Burger, Pizza, Sushi). Recomputed periodically. Only one store per cuisine holds it at any time.
Display
"Most Popular in Burger"
The cuisine name is dynamic — rendered at client side from the signal payload. Only shown when the store has a cuisine assigned and holds the #1 rank.
Eligibility
Sparse — max 1 per cuisine
Most stores will NOT be eligible. Guardrail metric: % of impressions showing the tag. If too low, the experiment may be underpowered — check before launch.
